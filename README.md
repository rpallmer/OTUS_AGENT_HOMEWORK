# OTUS_AGENT_HOMEWORK

**Самостоятельная работа по теме:** «Работа с фреймворками и агентами»

---

## 📋 Описание проекта

Этот проект представляет собой **демонстрацию построения AI-агента**, реализующего несколько современных паттернов и технологий в области генеративного ИИ и агентских систем.

---

## 🎯 Цели и задачи

Целью проекта является:

*   Показать **применение фреймворка LangGraph** для построения **агентских графов**.
*   Интегрировать **RAG (Retrieval-Augmented Generation)** с использованием фреймворка **LangChain** и оптимизированного **Retriever**.
*   Обеспечить **наблюдаемость (Observability)** процесса работы агента через **Langfuse** (трейсинг, дебаг, подсчёт токенов).
*   Продемонстрировать **интеграцию с внешними API**, в том числе через **CLI-интерфейсы** с использованием `openapi-to-cli`.

---

## 🧱 Паттерны и технологии

Проект демонстрирует следующие ключевые паттерны и технологии:

1.  **LangGraph:** Используется для **построения агентских схем** в виде ориентированных графов, позволяющих моделировать сложные логические цепочки, включая вызовы инструментов.
2.  **RAG (Retrieval-Augmented Generation):** Интеграция с **векторной базой данных** через **LangChain**, что позволяет агенту извлекать релевантную информацию из внешнего контекста (базы знаний) для генерации более точных ответов.
3.  **Langfuse:** Обеспечивает **мониторинг и трейсинг** выполнения цепочек и узлов графа, что критично для отладки и оптимизации производительности агента.
4.  **FastAPI / Внешние API / CLI:** Использование **FastAPI** для создания внутренних сервисов (например, `billing_api.py`) и интеграция с ними через **CLI-команды**, возможно, сгенерированные с помощью `openapi-to-cli`. Это показывает, как агент может взаимодействовать с внешними системами.

## 🏃‍♂️ Запуск и использование

1.  Запустить `Ollama` и убедиться, что модели `mistral:7b`, `bge-m3` загружены.
2.  Проверить `Qdrant` (через Ollama).
3.  Запустить `Langfuse` (если используется локально).
4.  Запустить `billing_api.py` .

    4.1.  **Запуск сервиса:**
        ```bash
        uvicorn billing_api:app --host 127.0.0.1 --port 8001 --reload
        ```
        Сервер запустится на `http://127.0.0.1:8001`.
        FastAPI автоматически сгенерирует и предоставит OpenAPIцию по адресу `http://127.0.0.1:8001/openapi.json`.

    4.2.  **Настройка (онбординг) `ocli`:**
        ```bash
        npx openapi-to-cli onboard \
          --api-base-url http://127.0.0.1:8001 \
          --openapi-spec http://127.0.0.1:8001/openapi.json
        ```
        Эта команда создаст профиль `default` и сгенерирует CLI команды на основе `/openapi.json`.

    4.3.  **Проверка команд:**
        Чтобы посмотреть возможные команды после шага 4.2:
        ```bash
        npx openapi-to-cli commands
        ```

5.  Запустить основной скрипт `main_graph.py`.

### Описание сервиса:

Сервис выполняет функции корпоративного чат-бота, который обрабатывает входные обращения от клиентов и формирует ответ.

Входные обращения модель классифицирует на группы:

1.  **Вопрос, не требующий для подготовки ответа взаимодействия с внешними системами** (например, биллинг).
2.  **Вопрос, требующий взаимодействия с внешней системой** (например, для получения данных по лицевому счёту).
3.  **Вопрос, не связанный с деятельностью компании**.

Для каждой группы выбирается соответствующий способ обработки:

*   **Для первой группы:** Подключается инструмент **RAG** (Retrieval-Augmented Generation) для поиска информации в локальной базе знаний.
*   **Для второй группы:** Подключается **агент**, взаимодействующий с внешней системой (через CLI команды).
*   **Для третьей группы:** Формируется стандартный ответ о несоответствии вопроса деятельности компании.

Основная логика работы сервиса определяется **системным промптом**, который передаётся в LLM вместе с запросом клиента.

Подключён **LangFuse** (`http://localhost:3000`) для мониторинга работы запросов к Ollama и инструментам (скриншоты приложены в папке `/data`).
